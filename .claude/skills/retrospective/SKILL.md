# レトロスペクティブスキル（機能・スプリント単位の振り返り）

## 使い方
「retrospective して」「この機能のレトロを書いて」「スプリントを振り返って」
「PRのまとめ/振り返りを作って」と言われたら使う。

## reflect スキルとの違い
- `/reflect` … 1セッションの**学びを `.claude/rules/` と `memory/` に反映**する（再発防止・ルール化）。
- `/retrospective`（これ）… **機能/スプリント/PR単位の成果と意思決定を文章化**する
  （KPT・メトリクス・設計判断の記録）。チーム共有・PR本文・Note素材に流用できる。
- 学び（ルール化すべき項目）が出たら、最後に `/reflect` へ渡す。

## 手順

### STEP 1: 対象範囲を決める
どこからどこまでを振り返るか（コミット範囲・ブランチ・期間）を確定する。
```bash
git log --oneline -15
git diff <base>..HEAD --stat   # 例: 機能着手前のコミット..HEAD
```

### STEP 2: KPT で整理する
| 区分 | 書くこと |
|------|---------|
| やったこと | 実装・変更の要約（ユーザー価値で書く） |
| Keep（良かった） | うまくいった進め方・判断 |
| Problem（課題） | つまずき・手戻り・未解決の負債 |
| Try（次の一手） | 次に試す改善・残タスク |

### STEP 3: 意思決定の記録
設計上の分岐と「なぜそれを選んだか」を残す
（plan mode の AskUserQuestion で決めた選択、トレードオフ、却下した案）。
あとで「なぜこうなっているか」を辿れるようにするのが目的。

### STEP 4: メトリクス
- 変更規模（`git diff --stat` の合計）、コミット数
- テスト数・型チェック結果（worker/web 双方）
- マイグレーション番号、デプロイ結果（CI conclusion）

### STEP 5: 出力
- `docs/retrospectives/YYYY-MM-DD-<topic>.md` に保存（任意。フォルダが無ければ作る）。
- PR 本文・Note 記事（note-writer）・Threads（threads-writer）の素材にも流用する。

### STEP 6: reflect へ橋渡し
Problem / 学びの中で**再発防止のルールにすべきもの**があれば `/reflect` を実行して
`.claude/rules/` ・ `memory/` に落とす。

## 出力テンプレート
```markdown
# レトロスペクティブ: <機能名>（YYYY-MM-DD）

## 範囲
<コミット範囲 / ブランチ / 期間>

## やったこと（ユーザー価値）
- ...

## Keep（良かった）
- ...

## Problem（課題）
- ...

## Try（次の一手）
- ...

## 主要な意思決定
- 決定: ... / 理由: ... / 却下した案: ...

## メトリクス
- 変更: +X / -Y（Nファイル）, コミット M 件
- テスト: worker N pass / web M pass, tsc clean
- マイグレーション: 8XX / デプロイ: success

## reflect へ回す学び
- ...（あれば /reflect）
```

## 注意
- コードは変更しない（記録・文書化のみ）。
- 事実に忠実に。失敗・手戻りも省略せず書く（次回の改善材料になる）。
- ルール化は `/retrospective` では行わない。必ず `/reflect` を通して人間の承認を得る。
