# 振り返りスキル（/reflect）

## 使い方

セッション終了時に呼ぶ。
「今日何を学んだか」「同じミスを繰り返さないためのルールは何か」を整理して
`.claude/rules/` と `memory/` に反映する。

```
「/reflect」
「今日の作業を振り返って」
「学んだことをルールに追加して」
```

---

## 手順

### STEP 1: 今日の変更を確認

```bash
git log --oneline -10
git diff HEAD~3..HEAD --stat
```

---

### STEP 2: 修正パターンを特定する

以下の観点で変更履歴を読む：

| 観点 | 確認すること |
|------|-------------|
| 修正が必要だった箇所 | 最初の実装で何が間違っていたか |
| 同じパターンが複数回出た | 繰り返し発生している問題はないか |
| ドキュメントとコードがずれた | 規約に従っていないコードがあるか |
| テストが後から必要になった | TDD で先に書くべきだったケースはないか |

---

### STEP 3: `.claude/rules/` の既存ルールと照合

```bash
ls .claude/rules/
cat .claude/rules/api-coding.md | grep "###"
```

今日発生したパターンが既存ルールでカバーされているか確認する。
カバーされていない新パターンがあれば追記案を作る。

**追記フォーマット：**

```markdown
### [問題の概要]（YYYY-MM-DD 追記）

症状: （何が起きたか）
原因: （なぜ起きたか）

❌ 誤（アンチパターン）
```code
// 悪い例
```

✅ 正（解決策）
```code
// 良い例
```

対処: （具体的に何をすればよいか）
```

---

### STEP 4: memory/ の更新確認

`~/.claude/projects/-Users-akihisa-line-harness-oss/memory/MEMORY.md` を確認し、
今日学んだことで記録すべきものがあれば追記を提案する。

保存すべき記憶の判断基準：
- 同じアドバイスを 2 回以上した → `feedback` タイプ
- 新機能フェーズが完了した → `project` タイプ
- ユーザーの好みや作業スタイルの新しい側面を発見 → `user` タイプ

---

### STEP 5: 振り返りサマリーを出力

```markdown
## 振り返りサマリー（YYYY-MM-DD）

### 今日やったこと
- ...

### 学んだこと・新しいルール候補
- ...（rules/ への追記案）

### memory/ への追記候補
- ...

### 次回への引き継ぎ
- ...
```

---

## 注意

- ルールへの追記は「提案」として出す。自動でファイルを更新しない
- 人間の確認・承認を経てから `git commit` する
- セッションで何も新しい学びがなかった場合は「追記なし」と明示して終了する
