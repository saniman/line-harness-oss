# LINE Harness — AI アーキテクチャ設計

> サブエージェント思想 × AI 自己改善システムへの発展計画

---

## ビジョン

LINE Harness は「AI が LINE を安全に操作するための基盤（ハーネス）」として設計された。
次のフェーズでは、**AI 自身がハーネスを改善し続ける自己発展型システム**へと進化させる。

```
現在                         目標
──────────────────────────────────────────────────────
人間 → Claude Code           人間 → オーケストレーター
  ↓ (単一エージェント)              ↓
  LINE Harness API             ドメイン専門エージェント群
                                 ↓
                               LINE Harness API
                                 ↓
                            [学習フィードバックループ]
                                 ↓
                            .claude/ 設定を自己更新
```

---

## 現状分析（2026-06-03 時点）

### 稼働中の AI インフラ

| 要素 | 現状 | 品質 |
|------|------|------|
| カスタムエージェント | 1 個（booking-debugger） | 反応型デバッグのみ |
| カスタムスキル | 3 個（debug / deploy / migrate） | 運用操作のみ |
| パーミッション設定 | settings.json に基本セット | 運用済み |
| メモリシステム | `memory/` に会話横断記憶 | 部分的に活用 |
| ルール | `rules/` に 4 ファイル | コーディング規約のみ |
| MCP サーバー | 25 ツール（packages/mcp-server） | 実装済み未活用 |

### 課題と機会

1. **エージェントが 1 個**：booking-debugger のみ。CRM 操作・開発支援・品質管理が未カバー
2. **スキルが運用特化**：`/debug`, `/deploy`, `/migrate` は手順の記述。分析・生成・学習がない
3. **フィードバックループなし**：失敗→学習→ルール更新のサイクルが存在しない
4. **エージェント間連携なし**：複雑タスクを複数エージェントに分解できない
5. **自己改善メカニズムなし**：`.claude/` 設定は人間が手動更新している

## アーキテクチャ概要

### 3 層構造

```
┌─────────────────────────────────────────────────────────┐
│  Layer 3: 自己改善ループ                                   │
│  rule-extractor / memory-curator / session-reflector      │
│  （成功・失敗パターンを .claude/ に自動反映）               │
├─────────────────────────────────────────────────────────┤
│  Layer 2: ドメイン専門エージェント                          │
│  event-manager / crm-analyst / campaign-orchestrator /    │
│  booking-debugger / tdd-agent / security-auditor          │
├─────────────────────────────────────────────────────────┤
│  Layer 1: ハーネス基盤（現状）                              │
│  Cloudflare Workers + D1 + LINE API + Stripe + GCal        │
└─────────────────────────────────────────────────────────┘
```

## 設計原則

1. **最小権限原則**：各エージェントは自分のドメインに必要な最小権限のみ持つ
2. **検証可能性**：エージェントの行動は必ずログ・テストで検証できる
3. **人間の監視**：自己改善は `docs/ai-architecture/` の変更を通じて人間がレビューできる
4. **段階的な自律性**：最初は提案のみ → 承認後に実行 → 実績が積み重なれば自律化
5. **後退可能性**：すべての変更は git で追跡・ロールバック可能
