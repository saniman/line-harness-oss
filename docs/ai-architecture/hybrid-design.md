# ハイブリッドエージェント設計

> プロセス指向とドメイン指向を状況に応じて使い分ける設計方針

---

## 設計の出発点：2つの軸

2つのアプローチは対立するものではなく、**直交する2軸**の上にある。

```
              不確実性が高い（新機能・未知領域）
                        ↑
        プロセス指向が有効
        （計画 → レビュー → 実装 → レビュー）
                        │
  ドメイン ─────────────┼───────────────── ドメイン
  （確立）              │                  （確立）
                        │
        ドメイン指向が有効
        （専門エージェントが直接対応）
                        │
                        ↓
              確実性が高い（既知のパターン・既存機能の改善）
```

**タスクの「ドメイン確立度」と「不確実性」の2軸**で、どちらのモードを使うかを決める。

---

## 2つのアプローチの特性まとめ

### プロセス指向（工程分割型）

エージェントの境界が「**何をしているか（フェーズ）**」で引かれる。

```
計画エージェント → 計画レビュー → 実装エージェント → 実装レビュー
```

| メリット | デメリット |
|---------|-----------|
| タスクの種類を問わず汎用的に使える | ハンドオフのたびにドメイン知識が劣化する |
| フェーズ境界が自然な承認ポイントになる | 単純なタスクでも全フェーズを通る無駄が生じる |
| フェーズ単体を独立して改善できる | 計画と実装の乖離（電話ゲーム問題）が起きやすい |
| CrewAI・AutoGen 等の事例が豊富 | ドメイン知識がどのエージェントにも蓄積されにくい |

### ドメイン指向（領域分割型）

エージェントの境界が「**何を知っているか（ドメイン知識）**」で引かれる。

```
予約管理エージェント（計画も実装もレビューも自分で判断）
顧客データ管理エージェント（同上）
```

| メリット | デメリット |
|---------|-----------|
| ドメイン知識が1つの場所に集約・蓄積される | 各エージェントがフルスタックになり定義が複雑になる |
| ハンドオフが少なくコンテキスト損失が少ない | 横断的な品質基準の統一が難しい |
| 人間の組織構造（専門家に聞く）と一致する | ドメインをまたぐタスクで主管が曖昧になる |
| ドメインが深くなるほどエージェントが賢くなる | 実績・エコシステムがプロセス指向より薄い |

### ralph-loop との関係

ralph-loop（plan → act → reflect → plan...）はプロセス指向の一形態。
「まだ手探り状態の手法」という評価は正確で、サブエージェントに十分なドメイン知識があれば、
計画ループを外部から強制するより内部で自律管理する方が品質が高くなる場合がある。

## ハイブリッド設計：3つのモード

### モード A：ドメインエージェント直接対応

**使う場面**：既知のドメイン内、既存機能の修正・改善、答えが自明なタスク

```
ユーザー依頼
    ↓
予約管理エージェント → 直接実装
（計画書なし・外部レビューなし）
```

**具体例**：
- 「キャンセル通知の文言を直してほしい」→ 予約管理エージェントが即対応
- 「友だちリストの表示件数を変えたい」→ 顧客データエージェントが即対応

---

### モード B：プロセスパイプライン起動

**使う場面**：新機能・スキーマ変更・複数ドメインをまたぐタスク・セキュリティに触れる変更

```
ユーザー依頼
    ↓
計画エージェント（何を作るか・影響範囲の整理）
    ↓
計画レビュー（セキュリティ・既存機能との干渉確認）
    ↓
各ドメインエージェントに委譲（実装）← ここでドメイン知識が活きる
    ↓
実装レビュー
```

**具体例**：
- 「顧客ランク機能を新規設計して実装してほしい」→ モード B
- 「payment テーブルに新カラムを追加してフローを変えたい」→ モード B

**重要**：ドメインエージェントはモード A でもモード B の実装フェーズでも使われる。
**使われ方が変わるだけで、エージェント自体は同じ。**

---

### モード C：ドメインエージェント内の内部ゲート

**使う場面**：中程度の複雑さ。外部にプロセスを立てるほどではないが、即実装もリスクがある

```
ドメインエージェントが受け取る
    ↓
「これは既知のパターンか？」を自己判断
    ├── YES → 直接実装（モード A と同じ）
    └── NO  → 内部で計画 → 自己レビュー → 実装の小ループを回す
```

外部にプロセスエージェントを立てるのではなく、エージェント定義の中に
「複雑なタスクのときは計画を立ててから動く」というルールを書く方法。

---

## モード切り替えのトリガー基準

| タスクの性質 | 向かうモード |
|------------|-------------|
| 既存ファイルの修正 | A（ドメイン直接） |
| バグ修正・文言変更 | A（ドメイン直接） |
| 既知パターンの繰り返し | A（ドメイン直接） |
| 中程度の複雑さ・判断が必要 | C（内部ゲート） |
| 新ファイルの作成 | B（プロセス） |
| DB スキーマ変更を伴う | B（プロセス） |
| 3 ドメイン以上にまたがる | B（プロセス） |
| セキュリティ・決済に触れる | B（プロセス） |
| 「何を作るべきか」が未確定 | B（プロセス） |

## アーキテクチャ全体図

```
ユーザーの依頼
    ↓
オーケストレーター（モードのルーティング判断）
    │
    ├─ モード A ──→ 予約管理エージェント
    │               顧客データ管理エージェント
    │               機能テストエージェント
    │               メッセージ最適化エージェント
    │
    ├─ モード C ──→ 上記エージェントが内部ゲートで自律判断
    │
    └─ モード B ──→ 計画エージェント
                       ↓
                     計画レビュー
                       ↓
                   各ドメインエージェント（実装）
                       ↓
                     実装レビュー
```

## 段階的な実装計画

**いきなり全部を設計しようとすると複雑になりすぎる。**
実績を積みながら段階的に足していく。

### 段階 1：ドメインエージェントのみ（モード A）

モード切り替えは人間が判断する。
「これは複雑そうだから計画してから」と自分で意識するだけで十分。

実装するもの：
- `event-manager`（予約管理）
- `crm-analyst`（顧客データ分析）
- `tdd-agent`（機能テスト）
- `message-optimizer`（メッセージ最適化）

完了条件：各エージェントが自分のドメインのタスクを一人でこなせる

---

### 段階 2：モード B をスキルとして定義（プロセスの半自動化）

よく使うプロセスパターンをスキルとして定義する。
`/new-feature` スキルが「計画→レビュー→委譲」の手順を呼び出す。

実装するもの：
- `/new-feature` スキル（新機能実装の手順書）
- `security-auditor` エージェント（計画レビュー担当）
- `migration-planner` エージェント（スキーマ変更の計画担当）

完了条件：「新機能を追加して」という依頼に対してプロセスが自動的に回る

---

### 段階 3：オーケストレーターによる自動ルーティング

オーケストレーターエージェントがトリガー条件を見てモードを自動選択する。

実装するもの：
- オーケストレーターエージェント（モードのルーティング判断）
- エージェント定義へのモード C ゲート追加（各ドメインエージェントの更新）

完了条件：ユーザーが意識せずとも適切なモードで動く

---

## 方向転換の判断基準

ドメイン指向に問題が出てきたサインを 3 つ定義しておく。
これらが重なったときにプロセス指向への部分的な転換を検討する。

1. **ドメインエージェントが「自分の担当外」の判断を頻繁に求めてくる**
   → ドメイン境界が正しく引けていない、またはオーケストレーターが必要

2. **ドメインをまたぐタスクでオーケストレーターが複雑になりすぎる**
   → モード B の比重を上げる、または計画エージェントを先に立てる

3. **エージェント定義の維持コストが実装コストを上回る**
   → 抽象度を下げてプロセス指向の汎用パイプラインに寄せる

## このプロジェクトでハイブリッドを選ぶ理由

LINE Harness は「Stripe の処理はこう、D1 のクセはこう、LINE 通知の落とし穴はこう」という
**確立したドメイン知識が価値の核心**にある。

- 既存機能の改善・修正が多い → モード A が頻繁に使われる
- 新 SP（スプリント）は不確実性が高い → モード B が必要
- 業務が深くなるほどエージェントが賢くなる → ドメイン指向が長期的に有利

プロセス指向は「汎用の道具」、ドメイン指向は「専門の道具」。
専門の道具を中心に置きつつ、汎用の道具を必要な場面で使うのがこの設計の基本姿勢。
