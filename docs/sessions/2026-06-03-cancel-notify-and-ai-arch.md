## 実施内容

### 1. キャンセル成功時の LINE Push 通知実装

**背景**
前セッション（2026-05-19）で Stripe 返金処理とキャンセル機能を実装済み。
今回、キャンセルが成功したことをユーザーの LINE 画面に通知する機能を追加した。

**変更ファイル**
- `apps/worker/src/services/events.ts`：`cancelEventBooking` の戻り値に `eventId` を追加
- `apps/worker/src/routes/events.ts`：キャンセルルートに LINE Flex メッセージ通知を追加
- `apps/worker/src/services/events.test.ts`：`eventId` のアサーションを 2 ケースに追加
- `apps/worker/src/routes/events.test.ts`：キャンセルルートのテストを 5 ケース追加

**通知の仕様**
```
ヘッダー（グレー #999999）: キャンセルが完了しました
ボディ:
  - イベント名（太字）
  - 日時（JST 変換）
  - 返金あり時のみ: 「返金処理を開始しました。5〜10 営業日かかる場合あり」
```

**設計上のポイント**

`cancelEventBooking` に `eventId` を返させることで、ルートでの余分な DB クエリを回避。
通知はベストエフォート（try/catch）— 通知失敗でキャンセル自体が失敗しない。

返金文言は Stripe の公式ドキュメントに基づき「処理を開始しました」表現に統一：
- 「完了しました」は誤り（Stripe はリクエストを送信した段階）
- 銀行・カード発行会社による処理には 5〜10 営業日かかることを明記
- 差戻し（元の支払い直後の返金）では個別入金が発生しない点も考慮

**テストケース（5 件追加）**
| ケース | 検証内容 |
|--------|----------|
| 正常系 | 200 + `refunded: false` |
| LINE 通知 | pushMessage が Flex で呼ばれる |
| 返金あり | 通知に「返金処理を開始しました」「5〜10 営業日」を含む |
| トークンなし | `LINE_CHANNEL_ACCESS_TOKEN` がなければ pushMessage を呼ばない |
| 異常系 | キャンセル済みは 400 + エラーメッセージ |

### 2. AI アーキテクチャ設計（調査・計画・ドキュメント）

**背景**
プロジェクトをハーネスエンジニアリングから「サブエージェント思想 × AI 自己改善システム」へ発展させる方針を決定。

**作成ドキュメント（`docs/ai-architecture/`）**

| ファイル | 内容 |
|--------|------|
| `README.md` | ビジョン・現状分析・3層アーキテクチャ概要 |
| `agents.md` | 7 エージェントのカタログと連携パターン 3 種 |
| `self-improvement.md` | hooks 設計・ルール自動更新・スキル昇格パターン |
| `roadmap.md` | Phase 1〜5 の具体的タスクと完了条件 |

**現状分析の主な発見**
- エージェント 1 個（booking-debugger）+ スキル 3 個（debug/deploy/migrate）のみ
- MCP サーバー（25 ツール）は実装済みだが AI 間連携・フィードバックループが未整備
- フック・自己改善メカニズムが存在しない

**設計した 7 エージェント**

| エージェント | 担当 | 優先度 |
|------------|------|--------|
| `tdd-agent` | Red-Green-Refactor の自律実行 | Phase 1 |
| `event-manager` | イベント〜Stripe〜LINE 通知の一連フロー | Phase 2 |
| `crm-analyst` | friends/tags/conversions の分析・レポート | Phase 2 |
| `security-auditor` | 新ルート追加時の OWASP チェック | Phase 3 |
| `migration-planner` | スキーマ変更の安全計画・実行 | Phase 3 |
| `rule-extractor` | 失敗→rules/ 更新案の生成 | Phase 4 |
| `memory-curator` | memory/ の整合性チェックと古い記憶の更新 | Phase 4 |

**自己改善の 5 メカニズム**
1. Claude Code Hooks（PostToolUse でテスト自動実行、Stop で振り返り促進）
2. ルール自己更新サイクル（同一バグ再発→ rules/ に追記）
3. スキル→エージェント昇格パターン（同じスキルが 5 回以上呼ばれたらエージェント化）
4. メモリ自動更新（月次で古い記憶を照合・更新）
5. エージェント評価ループ（実行ログ蓄積→定義の精緻化）
