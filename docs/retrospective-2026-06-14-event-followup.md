## 経緯（なぜこの3段階になったか）

### Phase 1: 機能決定（エージェント並列協議）
戦略系（crm-analyst）と技術系を並列起動 → 相互レビューで収束。
- 戦略系の当初推し: 案3（イベント後フォロー）。技術系の推し: 案A（自由文AI返信）。
- 相互レビューの結果、技術系が**実コードを調査**して「案3は配信基盤が既存で最小差分」と裏取り → 案3を第1弾に採択。
- 案A（パーソナルAI返信）は次フェーズ候補として保留。
- **学び**: 設計判断は「実コードを調べた側の根拠」が裁定の決め手になる。憶測ベースの推奨より、調査済みの主張を優先する。

### Phase 2: 開催日アンカー配信
ユーザーが実際に設定する段で「遅延(分)」が**申込時刻起点**だと気づき、「開催日の翌日10時」を実現できないと判明。
- `scenarios.delivery_mode`（'absolute_time'等）は**スキーマにあるだけで未実装**だった。
- 開催日アンカーを新規実装（`anchor_offset_days` / `send_time` / `friend_scenarios.anchor_at` + 純関数 `computeAnchoredDeliveryAt`）。

### Phase 3: 参加者一括登録（設計穴の修正）
「10時になっても届かない」という報告。本番D1を調査 → **確定参加者2名・シナリオ登録0件**。
- 原因: 自動登録は「予約確定の瞬間」にしか走らない。機能投入(6/13)より前に申し込んだ既存参加者は対象外。
- 対応: イベント詳細に「参加者を一括登録」ボタン（`POST /api/events/:id/enroll-participants`）を追加。

## 技術的な学び（再発防止）

### 1. テーブル再作成の前に実DBのカラム順を確認する
806（CHECK制約変更）でテーブル再作成が必要になった際、`schema.sql` は実DBと乖離していた
（scenarios に `line_account_id`/`delivery_mode` が無い、実DBでは `delivery_mode` が `line_account_id` より**前**）。
migration-planner が `SELECT sql FROM sqlite_master` で実DDLを確認し、明示カラムリストで INSERT...SELECT したため
データ列ズレを回避できた。schema.sql を信じて `SELECT *` していたら破損していた。

### 2. 共有型は2箇所（db / shared）にあり、shared は rebuild が要る
`ScenarioTriggerType` / `ScenarioStep` は `@line-crm/db` と `@line-crm/shared` の両方に定義されている。
db 側だけ変更すると web の tsc が古い型で落ちる。さらに web は shared の **dist** を参照するため、
`packages/shared` を `npm run build` しないと web の型チェックに反映されない（今回2回踏んだ）。

### 3. 「スキーマにある = 実装されている」ではない
`delivery_mode` カラムは存在したが `step-delivery.ts` は一切参照しておらず、配信は常に相対遅延だった。
機能の有無は列の存在ではなく、実際の処理コードを grep して確認する。

### 4. 編集前に「実際に描画されているコンポーネント」を確認する
`step-editor.tsx` を編集したが、これは**未使用**で、実UIは `scenario-detail-client.tsx` のインラインフォームだった
（画面スクリーンショットの「遅延(分)」で気づいた）。UI変更前に利用箇所を grep する。

## 設計上の学び

### トリガー型自動化は「新規発生分のみ」— 既存レコードへの遡及手段を併設する
「Xが起きたら自動でY」は発生時にしか走らない。今回「既存参加者に届かない」で表面化した。
新しい自動化を作るときは設計段階で次を確認する:
- (a) 既存（過去）レコードにも適用が必要か → 必要なら管理画面に一括適用アクションを併設
- (b) トリガーはグローバルか対象限定か（`event_booking` は全イベント共通。特定イベント限定は一括登録ボタンで実現）

### 時刻アンカーは指定時刻を厳守し、ステルス補正を外す
通常のステップ配信は 9-21時の枠補正＋jitter を掛けるが、開催日アンカーは「翌日10:00」を厳守したいので
これらを適用しない設計にした。配信タイミングの意図（厳密 vs ばらつき許容）でロジックを分ける。
